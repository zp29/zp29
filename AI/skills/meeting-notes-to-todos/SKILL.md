# 会议记录转待办

把原始记录转换为执行者拿到后即可理解和处理的任务清单。默认只返回清单，不解释整理过程。

## 工作流程

1. 通读全文，识别明确任务、BUG、待确认项、决定和背景信息。
2. 提取需要有人采取行动的内容；纯背景信息不单独创建任务，除非它影响验收标准。
3. 识别每项任务所属的业务模块、页面或对象，并将模块名写入任务开头。
4. 将一句中的多个独立动作拆开；将同一结果不可分割的条件保留在同一任务中。
5. 把口语、指代和省略表达改写成简短、具体、可执行的中文。
6. 检查是否遗漏否定词、条件、截止日期、字段名、文案、权限和状态变化。
7. 按原记录顺序输出只有一级的 Markdown 无序列表。

## 改写规则

- 每项使用动宾结构，例如“修改会员码标题为……”或“移除授权面积显示”。
- 保留产品中的准确名称、引号内文案、日期、编号和业务条件，不擅自扩充需求。
- 明确范围和条件，例如“电商经营单位”“审核通过后”“有编辑权限的操作人员”。
- BUG 应描述为待修复的预期结果；原记录未给出复现步骤时，不编造复现步骤或原因。
- 含糊但仍可执行的内容，按最保守含义改写。
- 存在影响执行的歧义或信息缺失时，列为 `待确认：...`，不要自行做产品决策。
- 重复事项合并，但保留不同模块下实际不同的任务。
- 不添加负责人、优先级、工时、截止时间或技术方案，除非原文明确提供。

## 输出格式

- 每条固定使用 `- 模块-具体任务` 格式，例如 `- 会员码-将标题修改为“……”`。
- 只输出一级列表，禁止父子任务、缩进列表或单独的模块标题。
- 同一模块有多个动作时，每个动作单独一行，并在每行重复模块名。
- 模块名与具体任务之间只使用一个半角连字符 `-`，不在其两侧添加空格。
- 不使用序号、表格、代码块、标题、前言、总结或完成状态。
- 用户明确要求复选框时，改用 `- [ ] 模块-具体任务`；明确指定其他格式时服从用户格式。

## 输出示例

输入：

```text
会员码标题改成“苏州市阳澄湖大闸蟹行业协会会员码”。电商不要显示专卖店基本信息。审核通过后有编辑权限的人还能修改和删除专卖店。
```

输出：

```text
- 会员码-将标题修改为“苏州市阳澄湖大闸蟹行业协会会员码”
- 电商经营单位-隐藏专卖店基本信息
- 专卖店后台管理-审核通过后，允许有编辑权限的操作人员修改专卖店信息
- 专卖店后台管理-审核通过后，允许有编辑权限的操作人员删除专卖店
```
